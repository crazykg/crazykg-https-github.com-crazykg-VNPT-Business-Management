## PHẦN 3: TỐI ƯU GIẢI THUẬT SINH KỲ THANH TOÁN

### Hiện trạng
Backend: Stored procedure sp_generate_contract_payments (migration 2026_02_27_210500) + PHP fallback trong V5MasterDataController.php (line 5571-5642)

Giải thuật hiện tại:
1. Đọc contract: payment_cycle, value, effective_date/sign_date, expiry_date
2. XÓA toàn bộ kỳ cũ của contract
3. Tính cycle_count = TIMESTAMPDIFF(MONTH, start, end) DIV interval_months + 1
4. Chia đều: base_amount = value / cycle_count, kỳ cuối nhận phần dư
5. Tạo N records payment_schedules với status PENDING

### Vấn đề hiện tại và đề xuất tối ưu

#### 3.1 Vấn đề: Xóa toàn bộ kỳ cũ khi sinh lại
- Hiện tại DELETE tất cả rồi INSERT mới → mất dữ liệu actual_paid đã nhập
- Đề xuất FE: hiện cảnh báo "Đã có X kỳ thanh toán với Y kỳ đã thu. Sinh lại sẽ xóa dữ liệu thu tiền. Bạn có chắc?"
- Đề xuất BE: option `preserve_paid=true` → chỉ xóa/tạo lại kỳ PENDING, giữ nguyên kỳ PAID/PARTIAL

#### 3.2 Vấn đề: Phân bổ đều chưa linh hoạt
- Hiện tại chia đều value / cycle_count cho mọi kỳ
- Đề xuất thêm chế độ phân bổ:
  + Chia đều (mặc định, giữ nguyên)
  + Thanh toán trước X% (ví dụ: kỳ 1 = 30%, còn lại chia đều)
  + Tùy chỉnh (user nhập số tiền cho từng kỳ sau khi sinh)
- FE: thêm dropdown "Cách phân bổ" trước nút "Sinh kỳ thanh toán"
- BE: thêm parameter allocation_mode và advance_percentage vào API generate-payments

#### 3.3 Vấn đề: Edge cases
- Contract có effective_date = expiry_date → cycle_count = 1 (OK nhưng nên validate)
- Payment cycle MONTHLY nhưng hợp đồng chỉ 15 ngày → tạo 1 kỳ (OK)
- Giá trị hợp đồng = 0 → sinh kỳ với expected_amount = 0 (nên skip hoặc cảnh báo)
- Đề xuất BE: thêm validation trước khi chạy procedure:
  + value > 0
  + expiry_date > effective_date
  + Nếu không thỏa → trả lỗi rõ ràng thay vì sinh kỳ rỗng

#### 3.4 SQL tối ưu stored procedure
- Thêm transaction wrapper: BEGIN...COMMIT để đảm bảo atomicity
- Thêm error handler: DECLARE EXIT HANDLER FOR SQLEXCEPTION → ROLLBACK
- Index: đảm bảo payment_schedules có index trên (contract_id, expected_date)

## PHẦN 5: ĐỀ XUẤT ỨNG DỤNG AI

### 5.1 Dự đoán dòng tiền (Cashflow Forecasting)
- Input: lịch sử thanh toán của tất cả hợp đồng (actual_paid_date vs expected_date)
- Output: xác suất thanh toán đúng hạn cho từng kỳ tương lai
- Logic: tính trung bình delay (ngày trễ) theo khách hàng, theo chu kỳ
- SQL gợi ý:
```sql
SELECT
  c.customer_id,
  AVG(DATEDIFF(ps.actual_paid_date, ps.expected_date)) as avg_delay_days,
  COUNT(CASE WHEN ps.status = 'OVERDUE' THEN 1 END) * 100.0 / COUNT(*) as overdue_rate
FROM payment_schedules ps
JOIN contracts c ON ps.contract_id = c.id
WHERE ps.status IN ('PAID', 'PARTIAL', 'OVERDUE')
GROUP BY c.customer_id;
```
- FE: hiển thị badge "Rủi ro cao/trung bình/thấp" bên cạnh mỗi kỳ thanh toán dựa trên avg_delay và overdue_rate của khách hàng đó

### 5.2 Gợi ý chu kỳ thanh toán tối ưu
- Khi user tạo hợp đồng mới, dựa trên:
  + Giá trị hợp đồng
  + Thời hạn hợp đồng
  + Lịch sử thanh toán của khách hàng đó
- AI gợi ý: "Khách hàng này thường thanh toán tốt theo quý. Đề xuất: Hàng quý"
- Logic BE: query lịch sử payment_schedules theo customer_id, tính cycle nào có overdue_rate thấp nhất → gợi ý

### 5.3 Cảnh báo quá hạn thông minh
- Cron job chạy hàng ngày: quét payment_schedules có expected_date < TODAY và status = PENDING → auto update status = OVERDUE
- Gửi notification/email cảnh báo trước X ngày (configurable): "Kỳ thanh toán HĐ ABC sẽ đến hạn sau 3 ngày"
- SQL:
```sql
-- Auto-update overdue
UPDATE payment_schedules
SET status = 'OVERDUE'
WHERE status = 'PENDING' AND expected_date < CURDATE();

-- Pre-warning query (3 ngày trước hạn)
SELECT ps.*, c.contract_code, c.contract_name, cu.customer_name
FROM payment_schedules ps
JOIN contracts c ON ps.contract_id = c.id
JOIN customers cu ON c.customer_id = cu.id
WHERE ps.status = 'PENDING'
  AND ps.expected_date BETWEEN CURDATE() AND DATE_ADD(CURDATE(), INTERVAL 3 DAY);
```

### 5.4 Phân tích xu hướng doanh thu
- Dashboard widget: biểu đồ dòng tiền dự kiến vs thực tế theo tháng
- SQL tổng hợp:
```sql
SELECT
  DATE_FORMAT(ps.expected_date, '%Y-%m') as month,
  SUM(ps.expected_amount) as expected_total,
  SUM(ps.actual_paid_amount) as actual_total,
  SUM(ps.expected_amount) - SUM(ps.actual_paid_amount) as gap
FROM payment_schedules ps
WHERE ps.expected_date BETWEEN '2026-01-01' AND '2026-12-31'
GROUP BY DATE_FORMAT(ps.expected_date, '%Y-%m')
ORDER BY month;
```
- FE: dùng chart library (recharts/chart.js) vẽ biểu đồ cột kép

### 5.5 Gợi ý giá trị hợp đồng từ dự án
- Khi tạo hợp đồng, AI phân tích:
  + Tổng giá trị hạng mục dự án (đã có: line 147-150 ContractModal.tsx)
  + So sánh với hợp đồng cùng khách hàng trước đó
  + Gợi ý: "Dự án này có X hạng mục, tổng giá trị Y VNĐ. Hợp đồng trước với KH này có giá trị trung bình Z VNĐ."
